## Summary

Phase 13.1 is a 90° rotation of the shipped Phase 13 `AeroPanelGroup`, not a new research domain. Every technical decision is already resolved. This research file's job is to VERIFY that the rotation is mechanical, surface the three places it is NOT trivial, and give the planner exact implementation guidance.

The three non-trivial issues are: (1) the rotated-Text measurement gotcha with `graphicsLayer`, (2) `BoxWithConstraints` axis switch from `maxHeight` to `maxWidth`, and (3) the `PanelGroupDivider` orientation flip from horizontal strip to vertical strip. Everything else (PanelDistribution.kt, aeroDragSplitter, AeroPanelGroupScope DSL, state model, animation, clamp) is verified orientation-agnostic and ports unchanged.

The public-API additive shape is: **add `orientation: Orientation = Orientation.Vertical` as a default parameter to the existing `AeroPanelGroup` function signature.** This is zero-breaking-change (no existing call site touches the parameter), follows the Kotlin default-arg convention rather than creating a second top-level function, and mirrors the `AeroSplitPane(orientation: AeroSplitOrientation = Horizontal)` precedent in this exact codebase. The risk of overload resolution breakage does not apply because this is a single function with one new defaulted parameter.

**Primary recommendation:** Refactor `AeroPanelGroup.kt` by extracting the layout core to an internal function `AeroPanelGroupImpl(orientation, ...)` in `internal/panelgroup/`, keeping `AeroPanelGroup(...)` as a thin public wrapper that forwards with `orientation = Orientation.Vertical` by default, and adding `orientation = Orientation.Horizontal` as an explicit caller option. The internal `PanelGroupDivider` composable receives the same `orientation` parameter and switches between horizontal (current) and vertical (new) strip rendering.

### Pattern 1: Additive Public API — Single Function with Default Orientation Parameter

**Precedent from this codebase:** `AeroSplitPane(orientation: AeroSplitOrientation = AeroSplitOrientation.Horizontal, ...)` — one public function, orientation selects the layout path inside.

**Use this exact shape:**

```kotlin
// AeroPanelGroup.kt — MODIFIED public signature (zero breaking change)
@Composable
public fun AeroPanelGroup(
    modifier: Modifier = Modifier,
    orientation: Orientation = Orientation.Vertical,   // NEW — default preserves all existing callers
    initiallyExpanded: Set<String>? = null,
    expandedKeys: Set<String>? = null,
    onExpandedChange: ((Set<String>) -> Unit)? = null,
    onLayoutChange: ((List<Float>) -> Unit)? = null,
    content: @Composable AeroPanelGroupScope.() -> Unit,
) {
    // ... collect scope ...
    AeroPanelGroupImpl(
        orientation = orientation,
        sections = sections,
        // ...
    )
}
```

**Why not a separate `AeroPanelGroupRow` function:** Two separate top-level functions means duplicating or triplicate-ing the public KDoc, both call `AeroPanelGroupImpl` with no behavioral difference, and the AeroSplitPane codebase precedent explicitly shows the single-function approach. The default parameter approach is Kotlin idiomatic and introduces zero overload-resolution risk because `Orientation` is an unambiguous type not used elsewhere in the parameter list.

**Existing call sites:** `LayoutSection.kt` line 187: `AeroPanelGroup(modifier = Modifier.fillMaxSize()) { ... }` — uses `modifier` named parameter; `orientation` is a new positional-second parameter with a default. Since existing callers use named `modifier =` and no other positional params before the DSL lambda, they are unaffected.

### Pattern 2: Orientation Branch Inside AeroPanelGroupImpl

The core decomposes into two orientation branches at exactly three points:

1. **`BoxWithConstraints` axis:** `totalPx = if (orientation == Orientation.Vertical) constraints.maxHeight.toFloat() else constraints.maxWidth.toFloat()`
2. **Layout container:** `Column` for Vertical, `Row` for Horizontal
3. **Section size modifier:** `.height(dp).fillMaxWidth()` for Vertical → `.width(dp).fillMaxHeight()` for Horizontal; divider: `fillMaxWidth().height(DIVIDER_THICKNESS)` → `fillMaxHeight().width(DIVIDER_THICKNESS)`

Everything else — state lists, expand/collapse logic, drag lambda, `rememberUpdatedState`, `isDragging`, `animateFloatAsState`, `onToggle`, `onLayoutChange` — is orientation-agnostic and runs identically.

```kotlin
// Inside AeroPanelGroupImpl:
val isHorizontal = orientation == Orientation.Horizontal

BoxWithConstraints(modifier = modifier.fillMaxSize()) {
    val totalPx = if (isHorizontal) constraints.maxWidth.toFloat() else constraints.maxHeight.toFloat()
    // ... all state, animation, drag logic unchanged ...

    if (isHorizontal) {
        Row(modifier = Modifier.fillMaxSize()) {
            // render sections as columns; dividers as vertical strips
        }
    } else {
        Column(modifier = Modifier.fillMaxSize()) {
            // existing vertical render path — ZERO CHANGE
        }
    }
}
```

### Pattern 3: Horizontal Section Layout — Header Strip + Content Column

Each horizontal section is a `Row` containing: a fixed-width header strip (36dp wide, `fillMaxHeight`) and a content area (`weight(1f)` or explicit `width(dp)`). Collapsed section = header strip only (no content Box rendered, same condition as vertical).

```kotlin
// Inside horizontal render path, per section:
Row(
    modifier = Modifier
        .fillMaxHeight()
        .then(
            if (i == lastExpandedIdx && isExpandedNow) Modifier.weight(1f)
            else Modifier.width(with(density) { animatedWidthPx.toDp() })
        )
) {
    // ── Header strip — full-height, 36dp wide ──────────────────────────────
    Column(
        modifier = Modifier
            .fillMaxHeight()
            .width(HEADER_WIDTH)          // HEADER_WIDTH = 36.dp (same constant, renamed for clarity)
            .glassPanel(cornerRadius = 8.dp)
            .clip(RoundedCornerShape(8.dp))
            .then(
                if (section.collapsible) Modifier.clickable { onToggle(i) }
                else Modifier
            ),
        horizontalAlignment = Alignment.CenterHorizontally,
    ) {
        // Chevron at top — CaretRight, rotates 0° (expanded ►) ↔ 180° (collapsed ◄)
        if (section.collapsible) {
            Icon(
                imageVector = AeroIcons.CaretRight,
                contentDescription = null,
                tint = AeroTheme.colors.onSurface,
                modifier = Modifier
                    .size(18.dp)
                    .padding(top = 8.dp)
                    .graphicsLayer { rotationZ = caretRotation },   // 0f↔180f
            )
        }

        // Title — rotated -90° so it reads bottom-to-top (IDE tool-window style)
        // PITFALL: graphicsLayer does NOT change layout measurement — Text still occupies
        // its UNROTATED width×height in the layout pass. Use weight(1f) in Column plus
        // wrapContentSize(unbounded=true) BEFORE graphicsLayer to prevent measurement
        // conflict. See graphicsLayer Pitfall #1 below.
        Box(
            modifier = Modifier
                .weight(1f)
                .fillMaxWidth(),
            contentAlignment = Alignment.Center,
        ) {
            Text(
                text = section.title,
                style = AeroTheme.typography.bodyMedium,
                color = AeroTheme.colors.onSurface,
                maxLines = 1,
                modifier = Modifier.graphicsLayer { rotationZ = -90f },
            )
        }

        // leadingIcon — below title (bottom of strip)
        section.leadingIcon?.let { icon ->
            Icon(
                imageVector = icon,
                contentDescription = null,
                tint = AeroTheme.colors.onSurface,
                modifier = Modifier.size(16.dp).padding(bottom = 8.dp),
            )
        }

        // headerActions — rendered in a non-bubbling inner Box
        section.headerActions?.let { actions ->
            Row(horizontalArrangement = Arrangement.Center) { actions() }
        }
    }

    // ── Content area — only when expanded (or animation still running) ──────
    if (isExpandedNow || animatedWidthPx > headerWidthPx) {
        Box(
            modifier = Modifier
                .fillMaxHeight()
                .then(
                    if (i == lastExpandedIdx && isExpandedNow) Modifier.weight(1f)
                    else Modifier.width(with(density) { (animatedWidthPx - headerWidthPx).toDp() })
                )
                .clipToBounds()
        ) { section.content() }
    }
}
```

### Pattern 4: Vertical PanelGroupDivider (Horizontal Orientation)

The `PanelGroupDivider` composable receives `orientation` and renders accordingly. For Horizontal orientation, the divider is a vertical strip (8dp wide, `fillMaxHeight`). Grip dots in a `Column` of 3 vertical dots (mirroring `SplitPaneDivider`'s `isHorizontal=true` path in `AeroSplitPane.kt` lines 203-216).

```kotlin
// PanelGroupDivider — orientation-aware
@Composable
private fun PanelGroupDivider(
    orientation: Orientation,
    onDrag: (deltaPx: Float) -> Unit,
    onDragEnd: () -> Unit,
    enabled: Boolean,
) {
    val isHorizontal = orientation == Orientation.Horizontal
    val modifier = if (isHorizontal)
        Modifier.fillMaxHeight().width(DIVIDER_THICKNESS)
    else
        Modifier.fillMaxWidth().height(DIVIDER_THICKNESS)

    Box(
        modifier = modifier
            .hoverable(interactionSource)
            .then(
                if (enabled) Modifier.aeroDragSplitter(
                    orientation = orientation,    // Horizontal → E_RESIZE; Vertical → N_RESIZE
                    onDrag = onDrag,
                    onDragEnd = onDragEnd,
                ) else Modifier
            )
            .background(if (hovered && enabled) colors.buttonHover else Color.Transparent),
        contentAlignment = Alignment.Center,
    ) {
        // 1dp visual line
        val lineModifier = if (isHorizontal)
            Modifier.fillMaxHeight().width(1.dp)
        else
            Modifier.fillMaxWidth().height(1.dp)
        Box(modifier = lineModifier.background(colors.borderDefault))

        // Grip dots — Column of 3 for horizontal divider (vertical grip visual)
        // matches AeroSplitPane.kt SplitPaneDivider isHorizontal=true branch (lines 203-216)
        if (enabled) {
            if (isHorizontal) {
                Column(modifier = Modifier.align(Alignment.Center), horizontalAlignment = Alignment.CenterHorizontally) {
                    repeat(3) { idx ->
                        if (idx > 0) Box(modifier = Modifier.height(4.dp))
                        Box(modifier = Modifier.size(3.dp).background(colors.labelText))
                    }
                }
            } else {
                Row(modifier = Modifier.align(Alignment.Center), verticalAlignment = Alignment.CenterVertically) {
                    repeat(3) { idx ->
                        if (idx > 0) Box(modifier = Modifier.width(4.dp))
                        Box(modifier = Modifier.size(3.dp).background(colors.labelText))
                    }
                }
            }
        }
    }
}
```

### Pattern 5: Chevron Angles for Horizontal (0° and 180°)

Vertical model: `animateFloatAsState(targetValue = if (expanded) 90f else 0f, ...)` — caret points down when expanded, right when collapsed.

Horizontal model: `animateFloatAsState(targetValue = if (expanded) 0f else 180f, ...)` — caret points right (►) when expanded (toward content), caret points left (◄) when collapsed (affordance of opening direction).

Both use the same `AeroIcons.CaretRight` glyph and `graphicsLayer { rotationZ = caretRotation }`.

### Anti-Patterns to Avoid

- **Duplicating `AeroPanelGroupScope` or `section(...)` DSL:** The scope is already orientation-agnostic. Do NOT fork it.
- **Using `Column` for the horizontal section render container:** Sections are side-by-side columns → must use `Row`.
- **Passing `constraints.maxHeight` for horizontal totalPx:** Use `constraints.maxWidth` in horizontal branch. Wrong axis = inverted size math.
- **Sizing header strip via `Modifier.height(36.dp)` in horizontal:** The strip is a column with `Modifier.width(HEADER_WIDTH).fillMaxHeight()`.
- **Using `Modifier.weight` for ALL sections in `Row`:** Same PNL-PITFALL-11 applies: give sections [0..N-2] explicit `.width(dp)`, last expanded section `.weight(1f)`.
- **Storing separate state for each orientation:** Single `sizePx`, `expandedState`, `lastExpandedFractionState` regardless of orientation. PanelDistribution is stateless and works on both.

## Common Pitfalls

### Pitfall 1: graphicsLayer{rotationZ} Does Not Change Measurement — Text Clips or Misaligns Inside 36dp Strip

**What goes wrong:** A `Text` with `graphicsLayer { rotationZ = -90f }` still MEASURES at its UNROTATED width × height. If the text string is 80dp wide, it measures 80dp wide and approximately `bodyMedium.lineHeight` tall. After rotation by -90°, it DRAWS as ~`lineHeight` wide × 80dp tall — but the layout slot it occupies is still 80dp × `lineHeight`. Inside a 36dp-wide header Column, the 80dp measured width exceeds the Column constraint. Compose wraps or clips the text.

**Why it happens:** `graphicsLayer` is a draw-phase transformation only. It does not participate in the Compose measurement pass. The parent layout measures the text at its intrinsic dimensions before rotation. Official docs: "Modifier.graphicsLayer does not change the measured size or placement of your composable, as it only affects the draw phase."

**How to avoid:** Place the `Text` inside a `Box(modifier = Modifier.weight(1f).fillMaxWidth())` with `contentAlignment = Alignment.Center`. The Box's cross-axis (width of the header Column = 36dp) constrains the Text's DRAWN width post-rotation. The Text's MEASURED height (line height ~20dp) fits comfortably in the 36dp width. Long titles get drawn outside the header strip's width boundary only if longer than the column height — which is fine because `graphicsLayer` does NOT clip by default. The text renders visually within the column's height (which is the section's full height, large) but its draw position will exceed the 36dp width into the content area. To prevent that visual leak, add `Modifier.clipToBounds()` to the Box wrapping the header strip, not to the Text itself.

**Canonical fitting pattern (CONFIRMED — zero custom Layout):**
```kotlin
// Header strip Column (36dp wide, fillMaxHeight):
Box(
    modifier = Modifier
        .weight(1f)           // fills remaining height of strip between chevron and icon
        .fillMaxWidth()
        .clipToBounds(),      // prevent rotated text draw from leaking into content area
    contentAlignment = Alignment.Center,
) {
    Text(
        text = section.title,
        style = AeroTheme.typography.bodyMedium,
        color = AeroTheme.colors.onSurface,
        maxLines = 1,
        softWrap = false,     // prevent soft-wrap on unrotated measurement
        modifier = Modifier.graphicsLayer { rotationZ = -90f },
    )
}
```

`softWrap = false` prevents Compose from wrapping the text in the narrow 36dp width BEFORE rotation. `clipToBounds()` on the parent Box prevents the drawn text from visually leaking outside the strip's bounds.

**`transformOrigin` is NOT needed** for centering: `contentAlignment = Alignment.Center` on the parent Box centers the text's logical bounds, and `graphicsLayer` rotates around the center by default (`TransformOrigin.Center = (0.5f, 0.5f)`). No `transformOrigin` adjustment is required.

**Confidence:** HIGH — `graphicsLayer` draw-only behavior confirmed via official Android docs. `softWrap=false` + `clipToBounds()` pattern is standard practice for this exact scenario.

### Pitfall 2: BoxWithConstraints Axis — Must Switch maxHeight → maxWidth

**What goes wrong:** If the horizontal branch reads `constraints.maxHeight.toFloat()` as `totalPx`, the available-for-expanded calculation uses the container's HEIGHT instead of WIDTH. All sections get distorted widths; drag resizes height instead of width.

**How to avoid:** Use `constraints.maxWidth.toFloat()` in the horizontal branch. The `AeroSplitPane.kt` lines 99-102 establish the exact pattern:
```kotlin
val totalPx = if (orientation == AeroSplitOrientation.Horizontal)
    constraints.maxWidth.toFloat()
else
    constraints.maxHeight.toFloat()
```
For `AeroPanelGroupImpl`, translate directly:
```kotlin
val totalPx = if (isHorizontal) constraints.maxWidth.toFloat() else constraints.maxHeight.toFloat()
```

### Pitfall 3: Modifier.Vertical/Horizontal Dimension Swap on Section and Divider

**What goes wrong:** In the horizontal branch, copying the vertical section modifier (`.fillMaxWidth().height(dp)`) gives a section that fills the full container width and has a fixed height — turning the Row into a series of stacked full-width boxes instead of side-by-side columns.

**How to avoid:** In the horizontal branch, all section layout modifiers swap axes:
- Vertical section: `Modifier.fillMaxWidth().height(animatedHeightDp)` → Horizontal: `Modifier.fillMaxHeight().width(animatedWidthDp)`
- Vertical last-expanded: `Modifier.fillMaxWidth().weight(1f)` → Horizontal: `Modifier.fillMaxHeight().weight(1f)`
- Vertical divider Box: `Modifier.fillMaxWidth().height(DIVIDER_THICKNESS)` → Horizontal: `Modifier.fillMaxHeight().width(DIVIDER_THICKNESS)`
- Vertical 1dp visual line: `Modifier.fillMaxWidth().height(1.dp)` → Horizontal: `Modifier.fillMaxHeight().width(1.dp)`

**Mental model:** Every `.fillMaxWidth()` becomes `.fillMaxHeight()` and every `.height(x)` becomes `.width(x)` in the horizontal branch.

### Pitfall 4: Chevron Angles — Horizontal Is 0°/180°, Not 0°/90°

**What goes wrong:** Copying the vertical chevron target angles (`0f` collapsed, `90f` expanded) to the horizontal variant makes the caret point DOWN when expanded instead of RIGHT.

**How to avoid:** In `AeroPanelGroupImpl`, derive `caretRotation` target conditionally:
```kotlin
val caretTarget = if (isHorizontal) {
    if (expandedState.getOrElse(i) { false }) 0f else 180f
} else {
    if (expandedState.getOrElse(i) { false }) 90f else 0f
}
val caretRotation by animateFloatAsState(
    targetValue = caretTarget,
    animationSpec = tween(durationMillis = 160, easing = FastOutSlowInEasing),
    label = "caret_${sections[i].key}",
)
```

### Pitfall 5: Public API Overload Resolution — Adding `orientation` as Second Parameter

**What goes wrong:** If `orientation` is added as the SECOND positional parameter after `modifier`, any existing caller that passes `modifier` positionally (not named) followed by anything else could shift. Example: `AeroPanelGroup(Modifier.fillMaxSize(), expandedKeys = ...) { }` — if `modifier` is passed positionally, the second argument is not `orientation` since it's named. In practice, all existing callers use trailing-lambda syntax and `modifier` is passed by name.

**How to avoid (LOW RISK, verified):** The existing call site `AeroPanelGroup(modifier = Modifier.fillMaxSize()) { ... }` uses a named `modifier =` argument. Adding `orientation: Orientation = Orientation.Vertical` as a new defaulted parameter anywhere in the parameter list does not affect this call. Kotlin default args require no overloads. The ONLY risk is if someone calls `AeroPanelGroup(Modifier.fillMaxSize(), someSet, ...)` positionally past the modifier — but `AeroPanelGroup` has no required parameters; all are defaulted. Existing callers in `LayoutSection.kt` use `modifier =` by name.

### Pitfall 6: `glassPanel` Gradient Direction in Vertical Strip

**What goes wrong:** `glassPanel` uses `Brush.verticalGradient` from top to bottom. For the vertical header strip (header is a full-height Column), the gradient runs from the top of the column to the bottom — which is visually correct (glass highlight at the top of the strip). However, for a narrow 36dp-wide strip this may look different from the horizontal header strip depending on the column height.

**How to avoid (LOW CONFIDENCE — verify visually):** No code change is needed; `glassPanel` simply draws on the column's background. The gradient will run top-to-bottom for the full height of the header strip, which is the correct Aero behavior. The three-theme sign-off (build step 8) is the visual gate.

### Pitfall 7: Regression on Vertical — `internal` Symbol Visibility and Scope DSL

**What goes wrong:** After extracting `AeroPanelGroupImpl` as an `internal` function or file, the `AeroPanelGroupScope` constructor is `internal constructor()`. Any moved code must remain within the same module (`:library`). Moving `AeroPanelGroupImpl` to `internal/panelgroup/` is safe because it is still in the same Gradle module.

**How to avoid:** Keep all refactored code in the `:library` module. Do NOT move anything to a new module. `AeroPanelGroupScope internal constructor()` is accessible from any file in `:library`. `PanelGroupDivider` is `private` in `AeroPanelGroup.kt` — after refactor it must be either private to the new internal file or moved to be a top-level `internal` composable in the same module.

## Open Questions

1. **`glassPanel` gradient orientation in tall vertical strip**
   - What we know: `glassPanel` uses `Brush.verticalGradient` from top to bottom. For a tall full-height column (the horizontal header strip), this gradient runs the full column height.
   - What's unclear: Whether a gradient spanning 360dp tall (typical column height) looks correct vs. the ~36dp tall horizontal header where the gradient is compressed. The Win7 Aero glass highlight is most effective over a short distance.
   - Recommendation: Verify visually during three-theme sign-off (build step 8). If the gradient looks washed out on a tall column, apply `graphicsLayer { rotationZ = -90f }` to the entire strip Box (not just the text) to make the gradient run left-to-right, matching the IDE tool-window aesthetic. This is a visual decision for the planner/implementer. LOW confidence on visual outcome without rendering.

2. **`headerActions` in horizontal strip — placement and direction**
   - What we know: In the vertical model, `headerActions` are at the right end of the horizontal header Row. In the horizontal model, the strip is a Column; `headerActions` should go below `leadingIcon` or above the chevron.
   - What's unclear: Which placement (below icon, above chevron, or at bottom of strip) is most ergonomic.
   - Recommendation: Place `headerActions` at the very bottom of the strip Column, below `leadingIcon` (matching VS Code's bottom-anchored action buttons in tool-window tabs). MEDIUM confidence — this is aesthetic judgment within Claude's discretion.
