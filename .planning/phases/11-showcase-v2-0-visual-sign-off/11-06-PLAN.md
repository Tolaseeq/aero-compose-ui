<!-- Root-cause diagnosis (verified against source 2026-06-18): -->

aeroDragSplitter (AeroDragSplitter.kt) computes delta as `cur.x - prev.x` across frames,
where `cur`/`prev` are `change.position` LOCAL to the 8dp hit-area Box. During a resize/divider
drag the hit-area Box itself relocates each frame (the column width / divider px changes), so
`prev` (last frame's box-local coord) and `cur` (this frame's box-local coord) are measured in
DIFFERENT coordinate frames. The leftover offset makes the delta smaller than the true cursor
motion (F15 reduced sensitivity) and the relocating consumed-drag visual doubles (F3 ghosting).
FIX: use the single-frame, frame-stable delta `change.positionChange()` (or
`change.position - change.previousPosition`) which is computed within ONE event in ONE coordinate
frame, instead of remembering `prev` across loop iterations.

AeroRangeSlider (AeroRangeSlider.kt) — the `pointerInput(enabled, valueRange, steps)` lambda
captures `value` ONCE at setup; the inner drag loop calls
`applyThumbMove(value, thumb, xToValue(change.position.x, ...), ...)` reading that STALE captured
`value`. `applyThumbMove` carries the non-dragged endpoint over from `current` — but `current` is
the stale value, so the other thumb snaps to its drag-start value every emit (F9). The HSV-square
and hue-slider are absolute-position pickers (`position.x / width`) so they do not suffer F9, but
they share the slow-follow class if any cross-frame `prev` is used.
FIX: read the live range via `val currentValue by rememberUpdatedState(value)` and call
`applyThumbMove(currentValue, ...)` inside the loop, so the non-dragged endpoint always reflects
the latest committed value.

Canonical drag pattern (keep): `awaitPointerEventScope { awaitFirstDown(...); loop { awaitPointerEvent(); if (!pressed) end; consume } }` — first-pixel response, no touchSlop. Do NOT
reintroduce detectDragGestures (banned v2.0-wide, PITFALL-03).
