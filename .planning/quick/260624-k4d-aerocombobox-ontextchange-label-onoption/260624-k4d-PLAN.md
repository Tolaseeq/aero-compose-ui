<objective>
Fix two coupled bugs in `AeroComboBox` (library bug â€” consumer usage is correct):

1. **Text overwrite after select.** The `onClick` handler calls `onOptionSelect(opt)` and then `onTextChange(opt)`. A consumer that clears the field inside `onOptionSelect` (e.g. command palette: `query = ""`) has its clear stomped by the trailing `onTextChange(opt)`.
2. **Popup re-opens on exact full-match.** `LaunchedEffect(focused, text)` recomputes `expanded` purely from focus + match existence. After a selection leaves the full label in the field, the effect re-fires (text changed) and the popup re-opens collapsed to the single exact match.

Purpose: Make `AeroComboBox` usable as a command-palette-style input (select â†’ field clears â†’ popup closes) WITHOUT breaking the existing default (select â†’ field shows the chosen label).
Output: A new pure-logic helper file, a JVM unit-test class locking both bugs closed plus no-regression cases, and the fixed `AeroComboBox` wired to the helpers with a new backward-compatible `clearOnSelect` parameter.
</objective>

<design_decision>
**Chosen approach: additive `clearOnSelect: Boolean = false` flag (NOT removing `onTextChange` after select).**

Why this over simply deleting the trailing `onTextChange(opt)`:
- Removing it unconditionally would change behavior for EVERY existing consumer who relies on the field displaying the selected label after a click (the common "type-ahead select" pattern). That is a silent breaking change.
- A `clearOnSelect = false` default is fully backward-compatible: existing callers behave identically. The command-palette caller opts in with `clearOnSelect = true`.
- It keeps the library in control of a single, predictable post-select text value instead of guessing whether the consumer cleared the field.

Post-select text rule (`textAfterSelect`):
- `clearOnSelect = false` â†’ return `opt` (current behavior; field shows the chosen label).
- `clearOnSelect = true`  â†’ return `""` (field clears; suits consumers whose `onOptionSelect` consumes the value, e.g. adds a command and resets the query).

Popup re-open rule (`shouldAutoOpen`) â€” fixes BUG 2 independently of `clearOnSelect`:
- A `justSelected` latch suppresses exactly one auto-open pass triggered by the selection's own text write, so the popup stays closed right after a click regardless of what text remains.
- Normal type-to-filter is untouched: when the user types, `justSelected` is false and the popup opens whenever focused and at least one option matches.
</design_decision>
